mc-customeronboarding-system-api-main:

Flow Functionality:

This flow serves as the primary entry point for the Customer Onboarding System API. It exposes the REST endpoints under /api/* and delegates incoming requests to the appropriate implementation flow using APIKit routing based on the RAML specification. It also manages the HTTP response by returning the appropriate status code and response headers generated by the downstream resource flows. This flow acts as the central gateway for all customer-related operations, ensuring that every request is routed according to the API contract.

Error Handling:

The flow implements APIKit-level error handling to provide standardized REST responses for invalid API requests. Client-side errors such as invalid requests, unsupported HTTP methods, unsupported media types, invalid resource paths, and unacceptable content types are intercepted and converted into meaningful JSON responses with the appropriate HTTP status codes. This prevents internal Mule exceptions from being exposed to API consumers and ensures consistent error responses across the application.

mc-customeronboarding-system-api-console:

Flow Functionality:

This flow hosts the APIKit Console, allowing developers and testers to explore the API documentation and interactively invoke the available REST endpoints. It simplifies API testing by exposing the RAML documentation through a web-based interface.

Error Handling:

If the requested console resource is unavailable, the flow returns a standard 404 Resource Not Found response instead of exposing internal routing exceptions.

POST /customers:
Flow Functionality:

This flow is responsible for onboarding new customers into the system. Before persisting data, it validates critical business fields such as the customer's email address and phone number to ensure data quality. After successful validation, the customer information is inserted into the Customer database and a standardized success response is returned. By validating the request before database insertion, the flow prevents invalid customer records from entering the system and maintains data integrity.

Error Handling:

The flow contains dedicated business-level error handling to provide meaningful responses for expected validation failures.

Validation errors such as an invalid email address or an incorrectly formatted phone number are captured using the Validation module and returned as user-friendly messages without exposing technical exception details.

Database insertion failures caused by duplicate Customer IDs are intercepted and converted into a 409 Conflict response indicating that the customer already exists. This prevents raw SQL exceptions from being returned to the client and clearly communicates the business reason for the failure.

GET /customers:
Flow Functionality:

This flow retrieves the complete list of customer records stored in the database. It acts as the customer retrieval service for the application by querying the Customer table and returning the results in a standardized API response containing metadata such as timestamp, API name, and operation status. It provides consumers with a consistent response structure regardless of the number of customer records available.

Error Handling:

The flow currently does not implement flow-level exception handling. Any database connectivity or SQL execution errors are propagated to the application-level error handler. This approach allows unexpected failures to be reported while relying on the application's centralized error handling mechanism.

GET /customers/{customerId}
Flow Functionality:

This flow retrieves a specific customer based on the Customer ID provided in the request URL. It performs a database lookup and returns the matching customer information within the application's standard response format. This endpoint enables consumers to retrieve individual customer details without querying the complete customer dataset.

Error Handling:

The flow currently relies on the default application error handling for unexpected database failures. If no matching customer exists, the current implementation returns the first element of the query result, which may be null. Business validation for handling missing customers has not been implemented explicitly.

PUT /customers/{customerId}
Flow Functionality:

This flow updates an existing customer's information in the database. The incoming request payload is preserved in a variable and used to execute the database update operation. Once the update completes successfully, the API returns a confirmation message indicating that the customer information has been updated. This flow enables modification of customer information while preserving the existing Customer ID.

Error Handling:

The flow currently does not contain dedicated flow-level exception handling. Database update failures, invalid SQL execution, or connectivity issues are propagated to the application's global error handling mechanism.

DELETE /customers/{customerId}
Flow Functionality:

This flow removes a customer record from the database using the Customer ID supplied in the request. Instead of assuming a successful deletion, it evaluates the number of affected rows returned by the database operation to determine whether a customer was actually removed. This allows the API to distinguish between successful deletions and requests for non-existing customers.

Error Handling:

Business-level validation is implemented after the delete operation by checking the affected row count. If a record is deleted successfully, the flow returns a success response. If no matching customer exists, it returns a business response indicating that the customer was not found. Unexpected database exceptions continue to propagate to the application-level error handler.

Employee Synchronization Flow (/salesforce)
Flow Functionality:

This flow implements a file-based Salesforce integration. Employee information is retrieved from an SFTP server and stored temporarily before being upserted into Salesforce using the Employee External ID. Using the upsert operation allows the application to create new employee records while updating existing ones without creating duplicates. After successful processing, the original employee file is archived locally to preserve a copy of the processed data for audit and recovery purposes.

Error Handling:

The flow currently depends on connector-level exception propagation. Failures related to SFTP connectivity, Salesforce authentication, file writing, or data transformation are not handled locally and are propagated to the application's default error handling mechanism.

Fetch Employee Flow (/fetchEmployees)
Flow Functionality:

This flow synchronizes employee information from Salesforce into the local database. Employee records meeting the specified business criteria are queried from Salesforce, transformed into the application's database structure, and inserted into the Employee table. The flow provides a synchronization mechanism between Salesforce and the local relational database while maintaining a consistent response containing the processed employee data.

Error Handling:

No explicit flow-level error handling has been implemented. Database insertion failures, Salesforce query failures, or duplicate record exceptions are propagated to the application-level error handler.

Scheduler with Scatter-Gather
Flow Functionality:

This scheduled flow performs concurrent data aggregation by executing multiple independent operations simultaneously using Scatter-Gather. It retrieves customer information from the database, employee information from the database, and employee data from Salesforce in parallel before combining all results into a single response. Using parallel execution reduces the overall processing time compared to sequential execution and improves the efficiency of scheduled synchronization activities.

Error Handling:

The Salesforce route is protected using a Try scope combined with an Until Successful scope. Temporary connectivity failures are automatically retried before the operation is considered unsuccessful. If all retry attempts fail, the exception is handled within the Try scope, allowing the remaining Scatter-Gather routes to complete successfully instead of terminating the entire scheduled process. This improves resilience while preventing a single system failure from affecting all parallel operations.

Customer Batch Processing
Flow Functionality:

This flow processes large customer datasets using MuleSoft Batch Processing. Customer records are read from a JSON file and processed independently, allowing the application to validate and insert each record without impacting the remaining records. This design makes the application suitable for high-volume customer onboarding scenarios where thousands of customer records must be processed efficiently. After execution, the batch framework automatically generates execution statistics summarizing the number of processed, successful, and failed records.

Error Handling:

The batch implementation performs record-level validation before inserting customer information into the database. Invalid customer records are identified and logged without interrupting the execution of the remaining records. Because Mule Batch isolates failures to individual records, one failed customer does not prevent the successful processing of the remaining dataset. The batch completion phase provides detailed execution metrics, making it easier to monitor data quality and identify failed records.
